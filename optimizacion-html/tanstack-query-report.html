🔬 Diagnóstico: El Problema Raíz

⚠️
Cache en Map<string> es ilusorio en producción serverless
En Vercel, cada invocación serverless puede ser una instancia diferente con memoria propia. El Map<string> que vivía en lib/cache-service.ts y lib/query-optimizations.ts nunca acierta un cache hit en producción bajo carga real.
Modelo mental: Vercel Serverless 
// Vercel puede crear una instancia diferente para cada request
│ Request A → Instancia 1 → Map { "orders_admin_all": [...] }  ← "guardado"
│ Request B → Instancia 2 → Map { }                              ← VACÍO siempre
│ Request C → Instancia 3 → Map { }                              ← VACÍO siempre

// La solución: cache en el CLIENTE (browser) — donde sí persiste
│ Usuario abre Admin  → TanStack  → fetch + guarda en QueryClient
│ Usuario cambia tab  → TanStack  → devuelve cache (0ms, 0 queries)
│ staleTime expira    → TanStack  → re-fetch en background

🔴 Anti-Patterns Corregidos

❌ Antes

  • ✕ 5 queries secuenciales en dashboard/page.tsx useEffect — sin cache, sin deduplicación
  • ✕ Re-fetch en cada evento Realtime sin debounce en orders/page.tsx — 10 órdenes = 10 fetches
  • ✕ Patrón N+1 en perfil/page.tsx: orders primero → luego order_items por cada orden
  • ✕ 60+ líneas de boilerplate por página: useState, loading, error, useEffect, fetchData
  • ✕ Map cache ilusorio — cache-service.ts y query-optimizations.ts nunca cachean en producción
  • ✕ Cambiar de tab en perfil → re-fetch completo aunque datos están "frescos"

✅ Después (TanStack Query v5)

  • ✓ useAdminStats() — 1 hook, Promise.all paralelo, cache 2 min en browser
  • ✓ useOrders() — Realtime invalida con 500ms debounce, no re-fetch directo
  • ✓ useUserOrders() — JOIN SQL directo, 1 query para orders + items
  • ✓ 2–4 líneas por componente: const { data, isLoading } = useHook()
  • ✓ Cache en QueryClient — instancia singleton en browser, sobrevive entre renders
  • ✓ Cambiar de tab → datos del cache instantáneamente (staleTime configurable)

⏱️ Configuración de staleTime por Dato

💡
¿Qué es staleTime?
Tiempo durante el cual TanStack considera los datos "frescos". Durante ese tiempo, si un componente pide la misma query, recibe los datos del cache instantáneamente (0ms, 0 fetch). Una vez expirado, el siguiente acceso lanza un re-fetch en background sin bloquear la UI.
Hook / Dato staleTime Justificación Invalidación
useAdminStats() 2 min Stats de conteo no cambian segundo a segundo Al crear/editar orden o suscripción
useOrders() 30 seg Alta frecuencia de cambio, Realtime como respaldo Supabase Realtime (500ms debounce)
useProducts() 10 min Catálogo cambia solo cuando admin edita Manual: invalidateProducts()
useProfile() 5 min Casi estático, cambia al guardar Mutation onSuccess
useUserOrders() 1 min Puede actualizarse por webhooks de pago Manual o webhook
useUserSubscriptions() 30 seg Sensible a cambios de estado de pago Mutation onSuccess

🏗️ Arquitectura de Cache

Flujo de datos con TanStack Query 
// 1. QueryClientProvider (una sola instancia en el browser)
  app/ClientLayout.tsx
  └─ QueryProviders               ← singleton, staleTime 60s default
     └─ CartProvider / Navbar / ...

// 2. Query Key Factory (strings tipados, sin magia)
  lib/query/keys.ts
  queryKeys.adminStats                     → ['admin-stats']
  queryKeys.orders.list({ page, status })   → ['orders','list',{...}]
  queryKeys.profile.orders(userId)          → ['profile',id,'orders']

// 3. Hooks (cache por queryKey)
useAdminStats() → staleTime 2min
useOrders()     → staleTime 30s  + Realtime debounce 500ms
useProfile()    → staleTime 5min
useProducts()   → staleTime 10min

// 4. Invalidación quirúrgica desde mutations
useMutation({
  onSuccess: () => {
    queryClient.invalidateQueries({ queryKey: queryKeys.orders.all })
    // Solo refresca Orders, no Profile ni Products
  }
})
